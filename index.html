The Problem

Containers give AI agents the keys to the kingdom

Shell access is the wrong abstraction for AI agents. They need to execute code, not operate environments.

01

Hallucination → Destruction

An AI that hallucinates can execute rm -rf / or pipe a malicious URL to bash. In a container, nothing stops it.

02

Prompt Injection → Exploitation

Attackers embed instructions in data. When agents process it with shell access, they become unwitting attack vectors.

03

Supply Chain Attacks

Runtime pip install means every package is an attack surface. One compromised dependency = full system access.

04

Data Exfiltration

Unrestricted network egress lets agents send sensitive data anywhere. Current sandboxes don't enforce allowlists.

05

Bloated Cold Starts

Gigabyte container images mean 30-120 second cold starts. Unacceptable for real-time agentic workloads.

⬡ The OpenCapsule Model

  • No shell exists — /bin/bash is structurally absent
  • No package manager at runtime — all deps are build-time only
  • No process spawning — single-process unikernel architecture
  • No arbitrary network — allowlist-only egress rules
  • 1-5MB images — sub-second cold starts from cache
  • KVM hardware isolation — not a software sandbox
  • E2E encrypted — per-job ephemeral keys
  • Fully open source — AGPL-3.0 licensed
Architecture

Three isolation layers. Zero shell access.

AI agents submit code manifests via HTTP. The system compiles, seals, and executes them in ephemeral MicroVMs.

Layer 3 — API Plane
HTTP REST API
Job submission, status polling, result delivery. E2E encrypted with XChaCha20-Poly1305.
Ed25519 Auth Axum REST
Layer 2 — Execution Plane
Firecracker MicroVMs + Unikraft Unikernels
Ephemeral builders compile Dockerfiles into sealed unikernels. Content-addressed cache provides 99% hit rate.
Firecracker Unikraft BLAKE3
Layer 1 — Isolation Plane
KVM Hardware Virtualization
Per-job MicroVMs with allowlist-only egress. No shell, no escape. The unikernel is the entire OS.

Features

Built for the agentic era

Every feature exists to make AI code execution safe, fast, and simple.

🔒

No Shell by Design

Unikernels don't have /bin/bash. Shell injection is structurally impossible, not just restricted.

⚡

Sub-Second Cold Starts

1-5MB unikernel images boot in 200-500ms. Content-addressed caching means near-instant starts for common stacks.

🛡️

KVM Hardware Isolation

Each job runs in its own Firecracker MicroVM with hardware virtualization. Not a software sandbox — real isolation.

🔐

E2E Encryption

XChaCha20-Poly1305 with per-job ephemeral keys. Code and results are encrypted in transit and at rest.

🌐

Allowlist-Only Egress

Agents can only reach endpoints you explicitly allow. No data exfiltration, no calling home to unknown servers.

📦

Multi-Runtime Support

Python 3.10+, Node.js 20+, and Bun 1.1. Run any language with statically-linked binaries baked in at build time.
